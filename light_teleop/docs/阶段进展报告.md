  -> TCP target 生成

当前没有引入 PyYAML，使用的是一个简单 yml 解析器，只支持当前配置需要的格式：

- 数字
- 布尔值 `true/false`
- `null`
- 简单列表，例如 `[-2, 3, 1]`

不支持复杂嵌套 YAML。

## 6. VR 控制状态机

当前控制逻辑：

```text
Grip 未按下：
  保持上一帧 target，不更新相对参考。

Grip 按下瞬间：
  锁定当前 VR 手柄 pose 为 VR reference。
  锁定当前机器人 TCP pose 为 robot reference。

Grip 按住：
  根据当前 VR pose 相对 reference 的变化生成 TCP target。

Grip 松开：
  释放 VR reference，保持最后 target。

X/A primary button：
  回到 home target，并重置 reference。
```

这里需要注意：当前绑定的是 TCP 点，不再只是 `left_arm_link9/right_arm_link9` 的 frame 原点。详见第 9 节。

## 7. 坐标与末端 frame

当前 Pink 控制 frame：

```text
left_ee_frame: left_arm_link9
right_ee_frame: right_arm_link9
```

`left_arm_link9/right_arm_link9` 是法兰盘/工具基座附近的 frame。

`left_arm_link10/right_arm_link10` 是相机/附属挂载点。当前不作为默认控制 frame。此前尝试过 `--stabilize-camera`，但现阶段配置里关闭了：

```yaml
stabilize_camera: false
```

原因是 wrist 调参阶段优先让 `link6/link7` 行为变清楚，不额外给 `link10` 姿态稳定任务施加约束。

VR 坐标轴映射仍为：

```yaml
axis_mapping: [-2, 3, 1]
```

左右手 tool orientation offset 来自：

```python
VrCoordinateTransformer.left_tool_offset()
VrCoordinateTransformer.right_tool_offset()
```

即左手 Z +90 度，右手 Z -90 度。

## 8. Wrist 自由度约定

当前调参中明确采用以下语义：

```text
link6 / joint6：腕部左右动
link7 / joint7：腕部上下动
```

从 URDF 看：

```text
left_joint6  axis xyz="0 1 0"
right_joint6 axis xyz="0 -1 0"
left_joint7  axis xyz="-1 0 0"
right_joint7 axis xyz="-1 0 0"
```

由于左右臂是镜像结构，符号可能需要单独调。当前先用一个全局符号解决：

```yaml
wrist_left_right_gain: -1.3
```

如果接手者发现左右方向又反了，优先改这个符号：

```yaml
wrist_left_right_gain: 1.3
```

## 9. Target 姿态模式

当前支持 4 种 target orientation mode：

```yaml
target_orientation_mode: wrist-decoupled
```

### 9.1 locked

```text
只跟随位置，姿态锁定在 Grip 绑定时的姿态。
```

优点：

- 稳。
- 不容易翻折。

缺点：

- `link6/link7` 基本不动。
- 不适合腕部调参。

### 9.2 swing

```text
只对齐工具轴方向，丢掉绕工具轴 roll。
```

优点：

- 比完整 VR 姿态稳定。
- 可以让腕部参与。

缺点：

- 有耦合。用户实测发现：往上翻腕后，左右灵敏度会变高。
- 原因是它基于工具轴方向做球面投影，上下倾斜会改变左右动作在工具轴上的投影。

### 9.3 vr

```text
完整跟随 VR 手柄旋转。
```

优点：

- 信息最完整。

缺点：

- 当前容易导致右手/腕部翻折。
- 暂时不作为默认。

### 9.4 wrist-decoupled

当前默认。

核心思想：

```text
把手柄相对旋转拆成两个固定腕部通道。

local X -> link7/joint7 上下
local Y -> link6/joint6 左右
local Z -> 丢掉
```

实现位置：

```text
light_teleop/light_teleop/vr_visual_demo.py
```

相关函数：

```python
wrist_decoupled_target_orientation(...)
apply_target_orientation_mode(...)
```

简化逻辑：

```python
reference_rotation = xyzw_to_rotation(transformer.robot_reference_orientation)
relative_rotation = reference_rotation.T @ target.rotation
rotation_vector = pin.log3(relative_rotation)

up_down = rotation_vector[0] * wrist_up_down_gain
left_right = rotation_vector[1] * wrist_left_right_gain

decoupled_rotation = pin.exp3([up_down, left_right, 0.0])
target_rotation = reference_rotation @ decoupled_rotation
```

当前默认参数：

```yaml
wrist_left_right_gain: -1.3
wrist_up_down_gain: 1.3
wrist_max_angle: 0.85
```

解释：

```text
wrist_left_right_gain:
  左右通道增益，主要影响 link6/joint6。当前是 -1.3，负号用于修正“往内拐”方向。

wrist_up_down_gain:
  上下通道增益，主要影响 link7/joint7。当前是 1.3。

wrist_max_angle:
  每个通道最大目标角度。当前 0.85 rad，约 49 度。
```

调参建议：

```text
左右不够灵活：
  wrist_left_right_gain: -1.5

左右方向反了：
  改符号，例如 -1.3 -> 1.3

上下不够灵活：
  wrist_up_down_gain: 1.5

翻到后面顶住：
  wrist_max_angle: 1.0

开始发飘：
  gain 从 1.3 降到 1.15，先不要动其它参数。
```

## 10. TCP 控制点

一个重要改动是：当前不再只控制 `link9` 原点，而是控制一个前移的 TCP 点。

当前配置：

```yaml
tcp_control_offset: 0.10
tcp_control_offset_axis: z
```

含义：

```text
受控点 = left/right_ee_frame 沿本地 z 轴前方 10 cm 的点
```

这样做的原因：

- 用户反馈只追 `link9` 原点时，腕部旋转会让实际工具末端绕 `link9` 走弧线。
- 视觉上会出现“腕部绑定点先到，真正末端不跟手”。
- 更符合直觉的目标是：让真正末端 TCP 点先到达/跟手。

当前链路：

```text
当前 link9 FK
  -> 加 tcp_control_offset 得到当前 TCP pose
  -> Grip 绑定时把 TCP pose 作为 robot reference
  -> VR 相对运动生成 TCP target
  -> TCP target 反算成 link9 frame target
  -> Pink 追 link9 frame target
  -> debug 绿/红方块显示 TCP target
```

相关函数：

```python
offset_target_position_local(...)
tcp_control_to_frame_target(...)
```

关键转换：

```python
tcp_translation = frame_translation + frame_rotation @ local_tcp_offset
frame_translation = tcp_translation - tcp_rotation @ local_tcp_offset
```

如果接手者发现 TCP 方向不对，优先调：

```yaml
tcp_control_offset: -0.10
```

或者改轴：

```yaml
tcp_control_offset_axis: x
```

当前之所以用 `z + 0.10`，是为了和 debug 方块设置保持一致：

```yaml
vr_debug_target_offset_axis: z
vr_debug_target_forward_offset: 0.10
```

## 11. Pink 权重与 posture profile

当前默认：

```yaml
posture_cost: 0.1
posture_cost_profile: joint6-priority
position_cost: 0.55
orientation_cost: 0.35
```

重要概念：

```text
posture cost 权重越大 -> 越想保持 home -> 越不愿意动
posture cost 权重越小 -> 越自由 -> 越愿意参与 IK
```

当前 `joint6-priority` 单臂 profile：

```python
[1.90, 1.90, 1.45, 1.10, 0.85, 0.005, 0.08]
```

乘上当前：

```yaml
posture_cost: 0.1
```

得到单臂实际权重：

```text
joint1: 0.190
joint2: 0.190
joint3: 0.145
joint4: 0.110
joint5: 0.085
joint6: 0.0005
joint7: 0.0080
```

这表示：

```text
joint6/link6 最自由，优先承担左右。
joint7/link7 仍然较自由，但不再像 joint6 那么自由。
前面大关节相对稳定。
```

已有 profile：

```text
uniform
front-flexible
wrist-balanced
wrist-priority
joint7-priority
joint6-priority
wrist-cross-priority
wrist-free
```

历史结论：

- `joint7-priority` 对上下较友好，但 `joint6` 被压得较硬，左右不灵活。
- `wrist-cross-priority` 同时放松 6/7，但左右仍不够明确。
- `joint6-priority` 当前更符合“向内抓取时优先让 link6 做左右”的需求。

## 12. VR Debug 可视化

开启：

```yaml
show_vr_debug: true
```

Meshcat marker 语义：

```text
青色/粉色小球：
  原始 VR controller 位置，显示在旁边的小坐标区域。

绿色方块：
  mapped-back target，限速前目标。

红色方块：
  实际发送给 Pink 的 target，经过 target_max_speed 后。

蓝色/黄色小球：
  Grip 绑定原点/anchor，用于看左右绑定点。
```

当前绿色方块做得更大且半透明，红色方块更小。这样两者重合时，绿色还能作为外壳看到。

当前 debug 方块显示 TCP target，而不是 `link9` 原点。这一点很重要：现在绿/红方块应该代表真正控制点。

相关配置：

```yaml
vr_debug_target_offset_axis: z
vr_debug_target_forward_offset: 0.10
```

这个只是可视化偏移。真正控制 TCP 的参数是：

```yaml
tcp_control_offset: 0.10
tcp_control_offset_axis: z
```

两者当前保持一致，避免看起来和控制点不一致。

## 13. Target 限速

当前：

```yaml
target_max_speed: 0.8
```

含义：

```text
每秒 target 平移最多 0.8 m。
```

现象：

- 绿色方块是限速前。
- 红色方块是限速后。
- 手柄大幅移动时，红色会追着绿色走。
- 如果 `target_max_speed: 0`，则关闭限速，红绿基本重合。

调试建议：

```text
怀疑限速导致不跟手：
  临时设 target_max_speed: 0

确认 target 稳定后再恢复：
  target_max_speed: 0.8
```

## 14. 当前用户体验结论

目前用户实测得到的结论：

- 双臂大部分工作希望向内抓取，所以 inward motion 是重点。
- `link7/joint7` 对上下翻腕已经能工作，但还需要一定灵敏度。
- `link6/joint6` 左右一开始不够灵活，后来通过 `joint6-priority` 和 `wrist-decoupled` 开始变灵活。
- 方向曾经反过，当前用 `wrist_left_right_gain: -1.3` 修正。
- `swing` 模式下，往上翻腕会让左右灵敏度变化，推测是工具轴对齐方式造成的几何耦合。
- 当前改为 `wrist-decoupled`，明确把 local Y 给 `link6` 左右、local X 给 `link7` 上下，减少上下和左右耦合。
- 最近一次调参把灵敏度增加为：

```yaml
wrist_left_right_gain: -1.3
wrist_up_down_gain: 1.3
wrist_max_angle: 0.85
```

如果太飘，优先回退为：

```yaml
wrist_left_right_gain: -1.15
wrist_up_down_gain: 1.15
wrist_max_angle: 0.85
```

## 15. 已知问题

### 15.1 左右臂镜像符号可能仍需单独处理

当前 `wrist_left_right_gain` 是全局参数，对左右臂都生效。

由于 URDF 中左右 `joint6` 轴一个是 `0 1 0`，一个是 `0 -1 0`，后续可能需要变成：

```yaml
left_wrist_left_right_gain: ...
right_wrist_left_right_gain: ...
```

目前尚未拆成左右独立参数。

### 15.2 wrist-decoupled 仍是工程近似

当前将 `pin.log3(relative_rotation)` 的 X/Y 分量直接映射到上下/左右通道：

```text
X -> joint7 上下
Y -> joint6 左右
Z -> 丢弃
```

这比 `swing` 更可控，但仍然依赖当前 frame 定义和手柄姿态映射。如果真实硬件手感仍不一致，需要继续做 per-arm / per-axis calibration。

### 15.3 TCP offset 仍需现场确认

当前：

```yaml
tcp_control_offset: 0.10
tcp_control_offset_axis: z
```

这是根据 debug 方块和当前观察做的工程假设。若真实末端点不在本地 z 轴正方向 10 cm，需要现场重新量或通过 Meshcat 观察改。

### 15.4 full VR orientation 仍可能翻折

`target_orientation_mode: vr` 暂不推荐。此前右手/腕部容易出现翻折。

### 15.5 当前仍未发布真机命令

`vr_visual_demo.py` 不发布关节命令，只做 Meshcat 可视化和 Pink IK。

## 17. 后续建议

优先级从高到低：

1. 现场继续验证 `wrist-decoupled + joint6-priority + TCP offset` 的手感。
2. 如果左右臂方向/灵敏度不一致，新增左右独立 wrist gain。
3. 如果 TCP 仍不跟手，重新确认真实 TCP 轴向和距离，优先调 `tcp_control_offset` 和 `tcp_control_offset_axis`。
4. 如果向内抓取需要更强 link6，继续增大 `abs(wrist_left_right_gain)`，不要先改 posture。
5. 如果上下翻腕不够，调 `wrist_up_down_gain` 或 `wrist_max_angle`。
6. 如果整体发飘，先降低 gain，再考虑提高 `posture_cost` 或降低 `orientation_cost`。
7. 后续若要接真机，需要单独实现 command publisher，并保留 dry-run / safety limit。
